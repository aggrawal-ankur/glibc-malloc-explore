# Chunk Description

Each malloc call returns the requested bytes of dynamic memory. Every portion of dynamic memory returned to a process has some bookkeeping attached to it by the allocator to manage it properly. This bookkeeping is kept in a structure called malloc_chunk.

A chunk holds metadata about a portion of memory that comes after it.

The layout of this chunk is as follows:
```c
struct malloc_chunk {

  INTERNAL_SIZE_T       mchunk_prev_size;
  INTERNAL_SIZE_T       mchunk_size;

  struct malloc_chunk*  fd;
  struct malloc_chunk*  bk;

  struct malloc_chunk*  fd_nextsize;
  struct malloc_chunk*  bk_nextsize;
};
```

The authors have acknowledged that this layout is misleading, but in my point of view, the layout is well-reasoned and the absence of that reasoning makes this layout "complicated".

This chunk layout is designed to keep the memory footprint minimum while allowing for a consistent design for three configuration of the metadata.

Briefly, we have two types of chunks: small chunks and large chunks. Together, they create three configurations of the metadata associated to the memory.

By definition, chunks are a piece of metadata associated to a portion of dynamic memory. But when we say "small and large chunks", it refers to the size of the memory these chunks have the metadata for.
  - The GNU allocator divides allocations in two sizes: small chunks and large chunks.
  - Starting from 0, up to a certain number, all requests are considered small. After that threshold, all requests are considered large.

---

The three configurations of this chunk metadata are:
  1. In-use chunks: chunks the process is actively using.
  2. Small free chunks: small size chunks the process has freed.
  2. Large free chunks: large size chunks the process has freed.

In-use chunks are self-managed and requires no extra bookkeeping. Free chunks, on the other hand, requires extra bookkeeping, because they can be used to service future malloc requests, so they must be managed in a way to be found easily when required.

mchunk_prev_size stores the size of the previous chunk and mchunk_size holds the size of the current chunk. Here, size means the total amount of payload memory returned to the process plus the chunk metadata.

**Note: INTERNAL_SIZE_T is not a data type in C. It's a tunable parameter in glibc which we will explore in an upcoming section. For now, consider it as `size_t`.**

GNU allocator manages free chunks via bins.
  - We have small bins for small chunks and large bins for large chunks.
  - These bins are made up of the "**circular doubly linked list**" data structure.

We have multiple small bins and large bins. Small bins manage free chunks of exact size classes, while large bins manage free chunks falling in specific size range. For example:
  - we have a small bin of size class 80 bytes. It contains free chunks which have a size of 80 bytes.
  - we have a large bin of size range [1024, 1088) bytes. It manages free chunks of sizes falling in that range.

There can be multiple chunks in a single bin, be it small or large. To manage links between them to traverse efficiently, we need a forward pointer and a backward pointer. The `fd/bk` members in the struct are for this purpose.

A large bin manages free chunks falling in a specific size range.
  - There can be two kinds free chunks: 1) free chunks falling in the specific range, and 2) free chunks falling in the specific range of same size.
  - For example, in the large bin [1024, 1088), we can have chunks of sizes 1024, 1056 and so on. Or we can have multiple free chunks of size 1056 bytes.

The fd/bk pointers are used to link all the free chunks in a large bin. The `fd_nextsize/bk_nextsize` pointers are used to point to the next/prev free chunks of the same size in a large bin.

This is the reasoning behind the existence of the six members in the malloc_chunk struct. The part that makes this layout misleading, or better, complicated, is the way this layout is used in reality.

We have six members here, pointers are 8-bytes wide and size_t is 8-bytes on 64-bit. 8*6 is 48 bytes. If we use `sizeof` to find the size of this struct, we will get 48 on 64-bit systems. But this chunk is never sized 48 bytes.

The size of this struct is situational and illusory. It varies for in-use chunks and free chunks.
  - Chunk overhead, which refers to the minimum size required to maintain the chunk metadata for a piece of dynamic memory is 16 bytes on 64-bit systems.
  - Those 16-bytes account for mchunk_prev_size and mchunk_size. The pointers are never maintained.
  - Once an in-use chunk is freed, we need space for fd/bk pointers because when this chunk will be binned, these pointers will get actively managed. That's the reason why the minimum allocatable size is 32 bytes on 64-bits.
  - The reason this size is partly illusory is that the metadata chunk for a piece of memory always start 8 bytes before the previous chunk's dynamic memory ends, and prevsize is managed only when the previous chunk is freed, not in-use.

A quick question that comes to our mind is that: "how is it possible? Isn't that violating struct integrity?"
  - No. Because the allocator is managing the layout of the memory itself.
  - Struct layout is managed by the compiler when allocated on stack. The allocator is allocating on dynamic memory, not static memory, it can allocate anywhere.
  - And it is not a problem that the current chunk starts bytes inside the boundary of the previous chunk's dynamic memory because that memory is managed only when the previous chunk is freed by the process.

The first and the last chunks are special chunks, and we will talk about them in an upcoming section.

The pointer to the dynamic memory that is returned to the process always point to the memory where `fd` is in the struct. This is not wrong because, in an in-use chunk, fd/bk/fd_nextsize/bk_nextsize are not required, so dereferencing is not defined for them. Therefore, we are not overwriting any memory and the chunk overhead remains minimum.

---

This is the annotation in malloc.c, where the author acknowledged the design as "misleading but accurate and necessary":
```
/*
  This struct declaration is misleading (but accurate and necessary).
  It declares a "view" into memory allowing access to necessary
  fields at known offsets from a given base. See explanation below.
*/
```

As we will explore the rest of the code, we will find that this design is "very well thought-out". It solves problems so efficiently with the least amount of overhead possible.

Therefore, the design is not misleading, it only lacks enough context that reveals the beauty of it.
